\section{Introducción}

Este informe describe el desarrollo de un sistema embebido capaz de generar, adquirir y clasificar señales en tiempo real utilizando una placa STM32 NUCLEO-U575ZI-Q. El proyecto combina técnicas de procesamiento digital de señales, sistemas operativos de tiempo real y aprendizaje automático embebido mediante NanoEdge AI Studio. El objetivo principal es identificar automáticamente diferentes formas de onda generadas por el propio sistema utilizando un modelo de clasificación ejecutado localmente en el microcontrolador. 

El codigo es abierto y está disponible en:

\begin{itemize}
	\item \url{https://github.com/ushikawa93/signal_classifier_with_rtos_stm32}.
\end{itemize}
 
\section{Objetivos}

\begin{itemize}
\item Generar señales sinusoidales, cuadradas y triangulares utilizando el DAC.
\item Adquirir las señales mediante el ADC.
\item Construir un dataset para entrenamiento.
\item Entrenar un clasificador utilizando NanoEdge AI Studio.

\item Utilizar FreeRTOS para coordinar las distintas tareas del sistema.

\section{Interface de usuario (tarea \texttt{StartUartTask})}

A través del puerto UART el sistema presenta al usuario un menú definido en el header \texttt{menú.h}. En este se le presentan las opciones:

\begin{itemize}
	\item 1 - Adquisición (captura de datos)
	\item 2 - Clasificación
	\item 3 - Configurar señales de DAC
	\item 4 - Configurar frecuencia de DAC
	\item 5 - Configurar nivel de ruido de DAC
\end{itemize}

La operación del menú y su derivación a las tareas correspondientes están controladas por la tarea \texttt{StartUartTask}.

\section{Generación de Señales (tarea \texttt{StartDacTask})}

La generación de señales se implementó utilizando:

\begin{itemize}
\item DAC interno (12 bits).
\item DMA en modo circular.
\item Temporizador como trigger.
\item Tablas precalculadas de muestras.
\item Generador de ruido interno del STM32.
\end{itemize}

Las formas de onda implementadas son:

\begin{itemize}
\item Sinusoidal.
\item Cuadrada.
\item Triangular.
\end{itemize}


\subsection{Control del DAC}

La tarea \texttt{StartDacTask} (definido en \texttt{app\_freertos.h}) junto con el header \texttt{signals.h} controlan la operación del DAC.

Esta tarea comienza agregándole ruido a las LU tables (\texttt{generateSignalsWithNoise(int)}) y luego lanza el DAC por DMA. Este esquema tiene la limitación de que el ruido se calcula solo en este punto, por lo que todos los ciclos de la señal tienen los mismos esquemas de ruido (es decir, el ruido es periódico). En un futuro podría mejorarse este punto recalculando las tablas en alguna interrupción que se de al finalizar el buffer del DMA. 

La transferencia memoria- DAC es disparada por el temporizador TIM2. Por defecto este timer esta configurado para generar una señal de 1 segundo de periodo, a partir de una LU table de 128 puntos (triggers cada 1/128 s). 

Esta tarea esta atenta a tres eventos que pueden disparase desde la UART que permiten modificar la forma de onda:

\begin{itemize}
	\item \texttt{DAC\_WAVE}: Cambio de la forma de onda.
	\item \texttt{DAC\_NOISE}: Cambia en nivel de ruido en la señal. Para ello detiene el DAC y vuelve a llamar a \texttt{generateSignalsWithNoise(int)}. El nivel de ruido se puede configurar de 0 a 4095 (cuentas del DAC).
	\item \texttt{DAC\_FREQ}: Cambia la frecuencia de actualización del DAC, llamando a la función \texttt{getTIMER\_ DAC(int)} para calcular el periodo del timer deseado a partir de la frecuencia de señal objetivo.
\end{itemize}


\section{Adquisición de Datos (tarea \texttt{StartAdcTask})}

La adquisición se realiza mediante el ADC utilizando DMA circular. Su ejecución esta definida en la tarea \texttt{StartAdcTask}. Esta configura la adquisición por DMA al principio de la operación y luego queda esperando eventos que señalizan bloques de datos listos para procesar.

Para ello se implementó un buffer doble de 256 muestras para permitir el procesamiento simultáneo de los datos mientras continúa la adquisición. La transacción del DMA es disparada por el TIM15, configurado para funcionar a 100 Hz (frecuencia de muestreo).

\subsection{Half Transfer y Transfer Complete}

A medida que se llena el buffer del ADC se disparan dos interrupciones que le avisan a la tarea \texttt{StartAdcTask} que hay datos disponibles para procesar

\begin{itemize}
\item \textbf{Half Complete}. Las primeras 128 muestras del buffer están disponibles
\item \textbf{Complete}. Las segundas 128 muestras del buffer están disponibles.
\end{itemize}

En esta etapa del desarrollo del proyecto el procesamiento de los datos consiste simplemente en imprimir cierta cantidad de bloques a través del puerto UART cuando el usuario lo solicita.

\section{Entrenamiento con NanoEdge AI Studio}

Utilizando la plataforma se adquirieron señales de distintas frecuencias, sinusoidales, ondas cuadradas y triangulares, que fueron utilizadas para entrenar un modelo de clasificación en NanoEdge AI Studio. Para adaptar las señales al formato csv requerido por el software se utilizaron scripts disponibles en la carpeta \texttt{/utils} del proyecto. En esta carpeta también hay disponible un script de matlab para graficar segmentos de las señales obtenidas y verificar su integridad.

\begin{itemize}
	\item \textbf{Dataset utilizado:} Los datos específicos que se utilizaron para entrenar el modelo se encuentran en la carpeta \texttt{nanoEdgeAI/dat}.
	
	\item \textbf{Número de clases:} El modelo se configuró con tres clases: \textit{OndaSinusoidal}, \textit{OndaCuadrada} y \textit{OndaTriangular}.
	
	\item \textbf{Configuración del entrenamiento:} Se empleó la versión 5.1.1 de NanoEdge AI Studio sobre la plataforma NUCLEO-U575ZI-Q, con señales de 128 columnas por muestra. El dataset fue balanceado con 3 archivos de 60 muestras por archivo en cada clase (180 líneas por clase). La búsqueda de la librería óptima se realizó mediante el benchmark automático (\textit{autoML}) de la herramienta, evaluando 18199 librerías candidatas durante 43 minutos con la configuración por defecto.
	
	\item \textbf{Resultados obtenidos:} El benchmark finalizó seleccionando una librería con un \textit{Quality Index} de 99/100 puntos y una 
	\textit{Balanced Accuracy} del 100\%, con un consumo de memoria de 1.7 KB de RAM (incluyendo buffer de 0.6 KB) y 7.8 KB de Flash. La matriz de confusión sobre el conjunto de validación no presentó ningún error de clasificación entre las tres clases (36 muestras correctamente clasificadas por clase), tal como se muestra en la Figura~\ref{fig:nanoedge_resultado}.
\end{itemize}

\section{Implementación del Modelo en STM32}

El modelo generado fue integrado al firmware mediante las bibliotecas proporcionadas por NanoEdge AI Studio.

El modelo se inicializa antes de lanzar las tareas, mediante la función \texttt{neai\_classification\_ init()} provista en la librería.

Luego, dentro de la tarea que controla el UART en la librería, se agregó la clasificación de señales. En este punto, cuando es requerido por el usuario se utiliza la función \texttt{neai\_ classification()}, pasandole el contenido del buffer proveniente del ADC que este completo en ese momento (puede ser la primera o la segunda mitad del ADC\_BUF). Antes de utilizarse directamente las muestras se convierten de entero a punto flotante, por requerimientos de la librería.

Usando este esquema el sistema decide entre alguna de las tres ondas lo que esta muestreando en el ADC, y expone los resultados junto con el nivel de similaridad obtenido.

\section{Resultados}

Para evaluar la robustez del sistema de clasificación ante condiciones de ruido, se realizaron 5 repeticiones de un test sistemático que consistió en clasificar cada tipo de señal (sinusoidal, cuadrada y triangular) bajo niveles crecientes de ruido: 0, 100, 200, 500, 1000, 2000 y 3000 (en unidades DAC). La señal sinusoidal fue clasificada correctamente en el 100\% de los casos independientemente del nivel de ruido, manteniendo una confianza promedio superior al 98\% en todos los niveles evaluados. La señal triangular presentó un comportamiento igualmente robusto, con un único error de clasificación observado en el test 1 con ruido nulo, posiblemente debido a condiciones transitorias al inicio del ensayo.

La señal cuadrada resultó ser la más sensible al ruido. Para niveles de hasta 1000 se obtuvo una tasa de aciertos aceptable, mientras que a partir de ruido 2000 el modelo comenzó a confundirla con la onda triangular, y con ruido 3000 la clasificación fue sistemáticamente incorrecta en los 5 tests, siendo identificada como onda sinusoidal en 4 de ellos. Este comportamiento es esperable dado que con ruido suficientemente elevado los flancos abruptos característicos de la onda cuadrada se suavizan, asemejando el espectro de otras formas de onda.

En términos generales, el modelo demostró una alta capacidad de clasificación para niveles de ruido de hasta 1000, con una tasa de aciertos global del 93\% en ese rango. Por encima de ese umbral, el desempeño se degrada principalmente en la clase cuadrada.


\section{Conclusiones}

El proyecto permitió ganar experiencia con la herramienta NanoEdgeAI y el entorno de desarrollo de STM32. El resultado final integra distintas tecnologías dentro de una única plataforma embebida:

\begin{itemize}
\item STM32.
\item DMA.
\item FreeRTOS.
\item NanoEdge AI.
\item Comunicación UART.
\item Aplicaciones de escritorio en Python.
\end{itemize}


En la versión actual el sistema permite generar adquirir y clasificar señales utilizando recursos limitados de procesamiento y constituye un punto de partida sólido para el trabajo futuro en estas plataformas. El sistema todavía tiene muchos puntos a mejorar pensando en su aplicación futura a temas de monitoreo atmosférico, que se detallan en la siguiente sección.

\section{Trabajo Futuro}

\begin{itemize}
\item Mejorar el método de agregado de ruido del sistema.
\item Incorporar señales externas.
\item Agregar nuevas clases a la clasificación.
\item Investigar la causa de la baja robustez de la clase \textit{OndaCuadrada} ante niveles de ruido elevados. En este sentido puede considerarse reentrenar el sistema con datos ruidosos.
\item Entrenar un algoritmo de detección de anomalías en las señales.
\item Probar el sistema con datos reales de sensores.
\item Optimizar el consumo de RAM y Flash de la librería generada, evaluando el trade-off entre tamaño del modelo y exactitud.
\item Validar el sistema en un rango más amplio de frecuencias de muestreo y adquisición.
\end{itemize}
